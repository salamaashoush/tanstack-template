# AGENTS.md

Working notes for AI coding agents (and humans in a hurry).

Most of what follows is not style preference — it is a list of traps this codebase
has already fallen into. Each rule exists because breaking it produced a real bug.

---

## Commands

| Do this                  | Not this                                                            |
| ------------------------ | ------------------------------------------------------------------- |
| `bun run test`           | `bun test` — Bun's own runner. Ignores `vitest.config.ts` entirely. |
| `bun run build`          | `bun build` — Bun's own bundler. Not the app build.                 |
| `bunx shadcn@latest ...` | `npx`                                                               |

```sh
bun install          # also compiles i18n and bootstraps .env
bun run dev          # :3000
bun run lint         # oxlint (type-aware) + message-catalogue parity
bun run format:write # oxfmt
bun run type-check   # tsc --noEmit
bun run test         # vitest
bun run test:e2e     # playwright, builds and serves the production bundle
```

**Bun is pinned to canary.** `bun.lock` is `lockfileVersion: 2`, which bun 1.3.x
cannot parse. The Dockerfile and `.github/setup/action.yml` both pin canary; if you
change one, change all three.

Bun is the package manager and script runner. **Node still executes the tooling** —
vite, vitest and playwright all have `#!/usr/bin/env node` shebangs, and the
production server is Node. Both runtimes are required.

## Never edit these

| Path                   | Why                                                        |
| ---------------------- | ---------------------------------------------------------- |
| `src/routeTree.gen.ts` | Generated by the TanStack Start plugin on dev/build.       |
| `src/i18n/**`          | Generated by Paraglide from `messages/*.json`.             |
| `src/components/ui/**` | Vendored shadcn registry code. `shadcn add` overwrites it. |

To change a `ui/` component, prefer a wrapper you own. If you truly must fork one,
say so in the PR — the next `shadcn add` will silently revert you.

Adding a route? Run `bun run dev` once so the plugin regenerates the route tree,
otherwise `tsc` will not know the route exists.

## UI: Base UI, not Radix

shadcn made Base UI the default. The package is **`@base-ui/react`** (not
`@base-ui-components/react`, which was abandoned at RC). Radix is gone from this repo.

- `asChild` does not exist. Use **`render`**: `<Trigger render={<Button />} />`.
- **`Menu.GroupLabel` must live inside a `Menu.Group`.** A bare `DropdownMenuLabel`
  throws `MenuGroupContext is missing` and takes the whole page down via the error
  boundary. Radix allowed it; Base UI does not.
- Style hooks are `data-open:` / `data-closed:` / `data-checked:`, not
  `data-[state=open]:`. The variants come from `@import "shadcn/tailwind.css"` — do
  not remove that import.
- `Select`'s `onValueChange` gives you `string | null`, not `string`.
- Prefer controlled inputs (`value` + `onChange`). An uncontrolled Base UI `Input`
  outside a `Field` does not reliably fire `onChange`.

`form.tsx` is retired upstream. **Forms are `Field` + `FieldGroup` +
`@tanstack/react-form`.** Valibot schemas plug straight into `validators` via
Standard Schema — there is no resolver package.

## Localisation is not optional

Every user-visible string lives in `messages/en.json` **and** `messages/ar.json`.
`bun run lint` fails if they drift (`scripts/check-i18n.ts`), because inlang removed
its own `lint` command with no replacement.

```tsx
import * as m from "~/i18n/messages";
m.membersTitle(); // yes
<h1>Members</h1>; // no
```

**Arabic is RTL, so use logical properties only**: `ps-*`/`pe-*`, `ms-*`/`me-*`,
`start-*`/`end-*`, `text-start`/`text-end`. Never `pl-*`, `ml-*`, `left-*`,
`text-left`. `components.json` sets `"rtl": true` so `shadcn add` stays logical too.

Locale resolves cookie → `Accept-Language` → `baseLocale`. `setLocale()` writes the
cookie and reloads, so the next document is _server-rendered_ in the new locale.

## Server, sessions, data

- Server functions live in `src/server/*.ts`. **Do not name a file `*.server.ts`** —
  Start's import protection denies that pattern in the client bundle by filename,
  even for a legitimately RPC-bridged `createServerFn`.
- **Throw `Error`, never a bare object.** Server-fn errors serialize through seroval;
  `throw { message: "..." }` does not round-trip. Use `throw notFound()` for a
  missing resource so the route renders its `notFoundComponent` and SSR answers 404.
- **The `QueryClient` is per-request**, constructed inside `getRouter()`. A module
  singleton leaks cache across requests on the server.
- `beforeLoad` must return only _serializable_ data. Do not spread the whole context
  back out — the router merges it, and the `QueryClient` in there is not serializable.
- Keep business logic in `src/data/*.ts` as **pure functions with an injectable
  dataset**. That is what makes it unit-testable without a server.

### The `env.PROD` trap

`env.PROD` (the runtime valibot object) resolves to its schema default — `false` — on
the server. Gate build-time concerns on **`import.meta.env.PROD`**, which Vite
statically replaces and dead-code-eliminates. Getting this wrong once shipped the
devtools to production and crashed SSR.

Secrets go in the `private` block of `src/env.ts`. Anything `VITE_`-prefixed is
**public and shipped to the browser**.

## Lint and types

- **oxlint + oxfmt.** There is no ESLint and no Prettier. `eslint-disable` comments do
  nothing; do not add them. If a rule is genuinely wrong, scope it in `.oxlintrc.json`
  with a comment explaining why.
- `noUncheckedIndexedAccess` is **on**. `array[i]` is `T | undefined`. Handle it —
  don't cast it away.
- Type-aware lint is on (`oxlint --type-aware`), so `no-floating-promises` is enforced.
  `void` a promise only when you genuinely mean to ignore it.

## Before you claim it works

`bun run type-check` passing is not evidence that a feature works. The things that
actually break here — SSR, sessions, locale, hydration — are invisible to `tsc`.

Run `bun run test:e2e`. It builds and serves the **production** bundle across
Chromium, Firefox and WebKit. Every one of the following was caught there and by
nothing else:

- a `Secure` session cookie that made sign-in silently impossible in Safari,
- devtools crashing SSR in production,
- two controls sharing one accessible name.
